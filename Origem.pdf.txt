“Dificuldade da população em compreender rótulos e fazer escolhas alimentares mais
saudáveis.”
O problema social:
Consumo de ultraprocessados – IBGE e Fiocruz:
●
●
●
●

Ultraprocessados representam cerca de 20% das calorias consumidas no Brasil.
Consumo mais elevado entre jovens.
Consumo maior em áreas urbanas.
Associação com aumento de obesidade e doenças crônicas.

Obesidade e doenças crônicas – Ministério da Saúde (Vigitel):
●
●
●

Mais de 50% dos adultos brasileiros estão acima do peso.
Cerca de 20% apresentam obesidade.
Hipertensão e diabetes seguem em crescimento.

Compreensão de rótulos – Anvisa:
●
●
●

Parte significativa da população tem dificuldade em interpretar tabelas nutricionais.
Termos como “%VD”, “gorduras saturadas” e “valor energético” não são claros para
muitos consumidores.
Implementação de rotulagem frontal com símbolo de lupa para indicar alto teor de
açúcar, sódio e gordura saturada.

A proposta:
O Origem é um aplicativo criado para ajudar as pessoas a entenderem melhor os rótulos
dos alimentos no momento da compra. Ao escanear o código de barras, o app mostra de
forma simples informações como açúcar, sódio, calorias e gorduras, utilizando alertas
visuais que facilitam a decisão.
Além disso, o aplicativo terá uma aba com vídeos curtos e informativos produzidos por
profissionais da área da saúde, explicando como interpretar rótulos, os riscos do consumo
excessivo de certos nutrientes e orientações básicas sobre alimentação equilibrada.
A proposta é tornar a informação nutricional mais clara e acessível, ajudando o consumidor
a fazer escolhas mais conscientes.

O produto:
O produto será um aplicativo mobile, desenvolvido em Flutter, utilizando a linguagem Dart.
O app terá como principal função a leitura de código de barras pela câmera do celular,
fazendo a consulta das informações nutricionais em uma API e exibindo esses dados de
forma simples e fácil de entender. A estrutura do sistema será organizada em camadas
(interface, lógica e dados), facilitando a manutenção e futuras melhorias.
O desenvolvimento será feito de forma incremental, com versionamento de código e
implementação das funcionalidades ao longo das etapas do projeto.

PARTE III – Requisitos Funcionais
RF01 – Leitura de código de barras
O sistema deve permitir que o usuário utilize a câmera do celular para escanear o código
de barras de um produto alimentício.
RF02 – Consulta de informações nutricionais
O sistema deve consultar uma API externa para obter as informações nutricionais do
produto escaneado.
RF03 – Exibição simplificada das informações nutricionais de forma simples e fácil de
entender. O sistema deve apresentar ao usuário as informações principais do produto,
como:
●
●
●
●

calorias
açúcar
sódio
gorduras

RF04 – Alertas visuais de nutrientes
O sistema deve destacar visualmente quando um produto possuir alto teor de açúcar, sódio
ou gordura, utilizando cores ou ícones de alerta.
RF05 – Exibição do nome e imagem do produto
O sistema deve mostrar o nome e, quando disponível, a imagem do produto escaneado.
RF06 – Histórico de produtos escaneados
O sistema deve permitir que o usuário visualize os últimos produtos escaneados.
RF07 – Lista de produtos favoritos / em compra
O sistema deve permitir que o usuário salve produtos escaneados como parte de uma lista
de compras ou favoritos.

RF08 – Criação de lista de compras manual
O sistema deve permitir que o usuário crie e edite listas de compras manualmente.
RF09 – Associação de listas a mercados
O sistema deve permitir associar listas de compras a mercados específicos.
RF10 – Notificação por localização (geolocalização)
O sistema deve identificar quando o usuário chegar a um mercado e exibir
automaticamente a última lista de compras associada àquele local.
RF11 – Pesquisa manual de produtos
O sistema deve permitir que o usuário busque um produto manualmente pelo nome.
RF12 – Mensagem para produto não encontrado
O sistema deve informar ao usuário quando um produto escaneado não for encontrado na
base de dados.

PARTE IV – Requisitos Não Funcionais
RNF01 – Usabilidade
O aplicativo deve apresentar as informações nutricionais de forma simples, clara e fácil de
entender, utilizando cores, ícones e textos curtos.
RNF02 – Desempenho
O sistema deve apresentar as informações do produto em até 3 segundos após o
escaneamento do código de barras, considerando conexão com internet.
RNF03 – Compatibilidade
O aplicativo deve funcionar em dispositivos Android e iOS, pois será desenvolvido
utilizando Flutter.
RNF04 – Disponibilidade de conexão
O aplicativo deve informar ao usuário quando não houver conexão com a internet, pois a
consulta das informações depende de uma API externa.
RNF05 – Segurança de dados
O aplicativo não deve armazenar dados pessoais sensíveis dos usuários, garantindo maior
privacidade.
RNF06 – Interface responsiva
A interface deve se adaptar a diferentes tamanhos de tela de smartphones.
RNF07 – Manutenibilidade
O sistema deve possuir arquitetura em camadas (interface, lógica e dados) para facilitar
manutenção e futuras melhorias.

RNF08 – Uso de localização
O aplicativo deve utilizar a localização do usuário para identificar quando ele está em um
mercado específico, respeitando permissões e privacidade.
